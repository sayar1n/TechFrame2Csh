# Мини веб-служба и конвейер обработки запросов

## Модульная архитектура с внедрением зависимостей

### Описание решения

Разработано приложение, которое загружает модули расширения из папки `modules` согласно списку в `config.json`, проверяет их зависимости, строит корректный порядок запуска (топологическая сортировка) и инициализирует модули. Каждый модуль:

- Имеет **имя** (строка, совпадающая с именем файла).
- Содержит **список требуемых модулей** (массив имён).
- Реализует **два шага**:
  1. `registerServices(container)` – регистрация собственных сервисов в DI-контейнере.
  2. `initialize(container)` – инициализация модуля (может использовать сервисы других модулей).

Приложение построено так, что **ядро не знает о конкретных модулях** – оно оперирует только их контрактом. Добавление нового модуля сводится к созданию файла в `modules/` и добавлению его имени в `config.json`. Ядро (файлы в `core/`) не требует изменений.

## Структура проекта

```
project/
├── app.js                # Точка входа (читает конфиг, запускает ядро)
├── config.json           # Список модулей для загрузки
├── core/
│   ├── container.js      # DI-контейнер (регистрация и разрешение сервисов)
│   └── moduleLoader.js   # Загрузка модулей, проверка зависимостей, топологическая сортировка, запуск
├── modules/              # Папка с модулями расширения
│   ├── logger.js         # Модуль логирования (без зависимостей)
│   ├── report.js         # Модуль генерации отчётов (зависит от logger)
│   └── export.js         # Модуль экспорта в файл (зависит от report и logger)
└── tests/                # Набор проверок (запускаются отдельно)
    ├── test-order.js     # Проверка корректного порядка запуска
    ├── test-missing.js   # Проверка ошибки отсутствующего модуля
    ├── test-cycle.js     # Проверка ошибки циклических зависимостей
    └── test-di.js        # Проверка внедрения зависимостей через контейнер
```

## Инструкция запуска

### Требования

- [Node.js](https://nodejs.org/) версии 12 или выше.

### Шаги

1. **Скопируйте все файлы** в соответствии со структурой выше.
2. **Убедитесь, что в `config.json` перечислены нужные модули** (например, `["logger", "report", "export"]`).
3. **Запустите приложение**:
  ```bash
   node app.js
  ```
4. **Ожидаемый вывод** (пример для трёх модулей):
  ```
   Starting application...
   [LOG] Logger module initialized
   [LOG] Report module initialized
   [LOG] Export module initialized
   [LOG] Generating report
   [LOG] Exporting to order.json
   [FILE] Writing to order.json: Report: {"userId":42,"items":["apple","banana"]}
   Initialization completed. Order: logger -> report -> export
  ```

### Запуск проверок (тестов)

Каждый тест – отдельный скрипт. Выполните в корне проекта:

```bash
node tests/test-order.js      # Проверка порядка запуска
node tests/test-missing.js    # Проверка отсутствующего модуля
node tests/test-cycle.js      # Проверка циклических зависимостей
node tests/test-di.js         # Проверка внедрения зависимостей
```

Все тесты используют встроенный модуль `assert` и не требуют установки дополнительных пакетов.

## Демонстрационные сценарии с разными наборами модулей

### Сценарий 1: Все модули включены (`config.json` = `["logger", "report", "export"]`)

- Модули запускаются в порядке: `logger` → `report` → `export`.
- `logger` регистрирует сервис логирования.
- `report` использует логгер для генерации отчёта.
- `export` использует генератор отчётов и логгер, чтобы экспортировать данные в файл.
- В консоли видны сообщения логирования и запись в файл – поведение реально меняется.

### Сценарий 2: Только `logger` и `report` (`config.json` = `["logger", "report"]`)

- Модули запускаются: `logger` → `report`.
- `export` не загружается, поэтому экспорта не происходит.
- Приложение завершает работу без ошибок, но экспорт не выполняется (это ожидаемое поведение).

### Сценарий 3: Отсутствующий модуль (`config.json` = `["logger", "missingModule"]`)

- Приложение завершается с ошибкой:
  ```
  FATAL ERROR: Missing required module "missingModule" for module "missingModule"
  ```
- Сообщение понятно указывает, какого модуля не хватает.

### Сценарий 4: Циклические зависимости (тестовый набор модулей с циклом)

- Если среди модулей есть цикл (например, `A→B→C→A`), приложение останавливается с ошибкой:
  ```
  FATAL ERROR: Circular dependency detected among modules: A, B, C
  ```

## Пояснение архитектуры: ядро и модули

### Что относится к ядру?

- `**core/container.js**` – реализация DI-контейнера. Он хранит фабрики сервисов и предоставляет методы `register` и `resolve`.
- `**core/moduleLoader.js**` – логика загрузки модулей из файловой системы, проверка контракта, валидация зависимостей, топологическая сортировка, запуск этапов регистрации и инициализации.
- `**app.js**` – точка входа, которая читает конфигурацию, создаёт контейнер и вызывает загрузчик.

**Ядро не содержит** никаких знаний о конкретных модулях (например, о `logger`, `report` или `export`). Оно оперирует только абстрактным контрактом: у модуля есть имя, зависимости, методы `registerServices` и `initialize`.

### Что относится к модулям?

- Все файлы в папке `modules/` (например, `logger.js`, `report.js`, `export.js`).
- Каждый модуль экспортирует объект, полностью соответствующий контракту.
- Модули регистрируют свои сервисы в контейнере и через контейнер получают сервисы других модулей.

### Почему добавление нового модуля не требует редактирования логики запуска?

1. **Обнаружение модулей** происходит динамически: приложение читает имена из `config.json` и загружает соответствующие файлы из папки `modules`. Добавление нового имени в конфиг – единственное изменение.
2. **Порядок запуска** вычисляется автоматически на основе зависимостей, указанных в самом модуле. Никакой жёстко заданной последовательности в ядре нет.
3. **Регистрация и инициализация** вызываются у всех модулей единообразно через циклы. Ядро не вызывает конкретные методы `logger.init()` и т.п. – оно вызывает `mod.initialize(container)` для каждого загруженного модуля.
4. **DI-контейнер** позволяет модулям получать зависимости без изменения кода ядра. Новый модуль может просто объявить зависимость от существующего сервиса и использовать его через `container.resolve`.

Таким образом, ядро остаётся неизменным при добавлении, удалении или модификации модулей – это ключевой признак расширяемой архитектуры.
